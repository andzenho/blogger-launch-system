# /audience — портрет аудитории

Запускает агента `audience-analyst`. Это центральный артефакт Discovery — от профиля зависят оффер, стратегия, весь контент прогрева. Каждое последующее решение опирается на эту работу.

## Когда звать

- `research/project-digest.md` собран (запускали `/intake`)
- По возможности принесены публичные сигналы через `/scout`
- Определено поле `has_base` в `state.md` (yes / no)

## Методология

Агент работает по схеме Хормози — выбор денежного сегмента по **4 факторам**:

1. **Pain** — сила боли (насколько горит)
2. **Purchasing Power** — способность платить
3. **Easy to Target** — легко ли найти / достать
4. **Growing Market** — есть ли тренд, рост спроса

Сегмент-победитель становится основным. Под него пишется `data/audience-profile.md` с детальным разбором: демография, внутренний мир, боли, страхи, возражения, Dream Outcome, язык сегмента (дословные цитаты).

## Два прохода

Агент работает **в два прохода**. Ветка выбирается автоматически по `has_base`:

### Ветка A — `has_base = yes`

**Проход 1 (до касдева).** На вход: digest + public-signals. На выход:
- `research/audience-hypotheses.md` — все сегменты на рассмотрение
- `research/money-segments.md` — скоринг по 4 факторам, выбран победитель
- `research/castdev-questionnaire.md` — анкета для Google-формы
- `research/castdev-bonus-options.md` — варианты бонуса за заполнение
- `research/castdev-brief.md` — ТЗ для warmer'ов (что писать в постах касдева)

Дальше продюсер создаёт форму, прописывает её URL в `state.md` → `castdev_form_url`, запускает `/castdev` — 3–4 дня прогрева параллельно в TG+IG собирают отклики и интервью.

**Проход 2 (после касдева).** На вход: Проход 1 + касдев-ответы в `brief/public-signals/castdev-responses.md`. На выход:
- `data/audience-profile.md` — финальный профиль, где все утверждения помечены источниками (`verified_comments` / `expert_claim` / `hypothesis`)

### Ветка B — `has_base = no`

Оба прохода в одном запуске, без касдев-артефактов:
- `research/audience-hypotheses.md`
- `research/money-segments.md`
- `data/audience-profile.md` — с пометкой «без верификации» (все утверждения — `hypothesis` или `expert_claim`)

## Precheck

Агент не стартует, если:
- `research/project-digest.md` не существует — нет контекста эксперта
- `has_base = TBD` в state.md — агент возвращает `blocked`, продюсер уточняет

## Выход

- **После Прохода 1:** research-артефакты + ТЗ на касдев (если has_base=yes)
- **После Прохода 2:** `data/audience-profile.md` → статус `client-review`

## После завершения

1. Продюсер читает профиль сам, проверяет: совпадает ли с ощущением от эксперта, достаточно ли данных в каждом блоке
2. `/review` → `client-review` → эксперт подтверждает
3. `/review` → `done` → можно звать `/product`

**Важно:** пока audience-profile не `done`, запускать `/product` нельзя — оффер будет бить не по тому сегменту.

## Типичные ошибки

- Пропустить касдев при has_base=yes ради скорости — теряем verified_comments, финальный профиль превращается в набор гипотез
- Не пометить источники (`[verified_comments]` / `[expert_claim]` / `[hypothesis]`) — на client-review эксперт не понимает, где факт, а где догадка, и не может внятно согласовать
- Выбрать сегмент не по деньгам, а по «нравится» — воронка будет работать, а выручки не будет
- Пытаться работать со всеми сегментами сразу — размывает позиционирование и контент прогрева
