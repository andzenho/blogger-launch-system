# /landing — посадочные страницы

Запускает агента `sales-page-writer`. Лендинги — конверсионные точки воронки. От качества первого экрана (H1, подзаголовок, оффер-превью) зависит CTR в регистрацию / покупку. Экономия на лендингах съедает весь трафик-бюджет.

## Когда звать

В зависимости от режима — см. Precheck ниже. Обычно этот этап идёт параллельно с `/event` и `/tg`/`/ig`/`/reels` в фазе Preparation.

## 4 архетипа лендинга

Из `.claude/skills/sales-page-writing/`:

### 1. `registration` — регистрация на событие

Лендинг под регистрацию на вебинар / эфир / марафон.
Структура: хук-обещание → «что узнаешь» → Why-you (короткий об эксперте) → программа → социальное доказательство → форма регистрации → таймер/scarcity.
**Precheck:** `research/event-brief.md` готов (из launch-strategist Проход 2).

### 2. `offer` — продажа после события

Продающий лендинг флагманского оффера. Два подрежима:

- **`direct-sale`** — прямая продажа по кнопке «купить». Долгий лендинг в духе Grand Slam: Value Stack, гарантия, тарифы, бонусы, FAQ, отзывы, scarcity / urgency
- **`application`** — через форму заявки + прозвон от ОП. Короткий лендинг на выявление боли + квалификацию. Используется для премиум-оффера или дорогих тарифов

**Precheck:** `data/product-matrix.md` в `done`, событие прошло (или близко).
**Обязательно client-review** — это главная продающая точка.

### 3. `leadmagnet` — холодный трафик на PDF/чек-лист

Короткий лендинг под opt-in. Одно обещание, минимум текста, форма в первом экране.
Применяется в cold / hybrid.
**Precheck:** `data/leadmagnet-funnel.md` готов, `launch_model` = cold / hybrid.

### 4. `thank-you` — подтверждение после опта

Страница, куда попадает человек после регистрации / заявки / покупки.
Задача: снять тревогу, дать чёткие следующие шаги (что будет / когда / что делать сейчас), микро-оффер (если уместно — трипваер / upsell / подписка на канал).

### `adhoc` — правка существующего лендинга

Разовая задача: переписать первый экран, добавить блок, перевести в новый оффер.

## Precheck (для любого режима)

Всегда на месте:
- `data/audience-profile.md` — язык, боли, Dream Outcome (основа заголовков и блоков)
- `data/product-matrix.md` — оффер, тарифы, цена, гарантия
- `data/launch-strategy.md` — модель, окно продаж, scarcity-механика

Режим-специфичные precheck — см. выше.

## Вход

- audience-profile / product-matrix / launch-strategy
- `research/event-brief.md` (для `registration`)
- `data/leadmagnet-funnel.md` (для `leadmagnet`)
- `data/cases.md` — кейсы для социального доказательства
- `research/project-digest.md` — тон, позиционирование

## Выход

- `content/landings/<type>-<slug>.md` — текст лендинга блоками (H1, подзаголовок, по секциям, CTA)
- Заметки по визуалу (что ставить на первый экран, где нужны фото эксперта, где — демо продукта)
- Варианты для A/B (обычно 2 варианта H1 + первого экрана)

## После завершения

- **`registration`** — продюсер проверяет сам, собирает на платформе, запускает регистрацию одновременно с анонсом в прогреве
- **`offer`** — **client-review обязателен.** Эксперт читает, подтверждает оффер и цифры. После — собирается и запускается перед окном продаж
- **`leadmagnet`** — продюсер проверяет, собирает, запускает вместе с трафик-кампаниями
- **`thank-you`** — собирается параллельно с предыдущими

## Типичные ошибки

- Писать лендинг до утверждения `product-matrix.md` — потом приходится переделывать весь блок «что входит» и цены
- На `offer`-лендинге терять Value Stack — всё предложение сжимается до «курс стоит X рублей», и Hormozi-ценность не считывается
- На `registration`-лендинге перегружать первый экран — посетитель не понимает, на что его зовут, и уходит
- Игнорировать кейсы из `data/cases.md` — социальное доказательство пропадает, конверсия просаживается
- На `application`-лендинге забыть форму квалификации — ОП получает «мусорные» заявки и не тянет объём
