# Blogger Launch System

Мультиагентная система для запуска продуктов с блогерами-экспертами. Работает в **Claude Code** на **Claude Pro** (модели Sonnet + Haiku, без Opus).

Система автоматизирует путь от сырого брифа эксперта до готового контента под запуск: прогревы в Telegram и Instagram, сценарии Reels, вебинар, посадочные страницы.

**Ещё до подписания договора** с экспертом можно собрать короткую pre-sale стратегию через `/presale` — документ на одну страницу с окнами продаж, декомпозицией цели по выручке, тарифами, условиями сотрудничества и таймлайном в неделях. Помогает закрыть сделку и визуализировать план для потенциального клиента. Подробнее — в разделе [Pre-sale стратегия](#pre-sale-стратегия-до-подписания-клиента).

## Что это и что умеет

Это набор специализированных AI-агентов и методологий, которые работают **последовательно** над одним запуском блогера. Каждый агент отвечает за свой кусок работы и оставляет артефакт для следующего.

**На входе:**
- Анкета эксперта (client-intake)
- Транскрипт созвона с экспертом
- Публичные сигналы (посты, комменты, рилсы)

**На выходе:**
- Профиль аудитории с денежным сегментом (по методике Хормози)
- Флагманский оффер (Grand Slam Offer)
- Стратегия запуска с таймлайном и юнит-экономикой
- Сценарий вебинара / ВСЛ / эфира
- Пакет прогрева: 3–6 недель постов в TG + IG + Reels
- Посадочные страницы (registration / offer / leadmagnet / thank-you)

**Используемые методологии:** Russell Brunson (Perfect Webinar, DotCom Secrets), Alex Hormozi ($100M Offers, $100M Leads), Jon Benson / Stefan Georgi (VSL), Iman Gadzhi (style), русскоязычные школы прогрева.

## Pre-sale стратегия (до подписания клиента)

Отдельный навык системы — `/presale`. Используется **до** любой реальной работы над проектом, чтобы закрыть сделку с потенциальным экспертом. Документ на 1–2 экрана, который отправляется эксперту в мессенджер или Notion.

### Что в него входит

- **Точка А** — краткая диагностика блогера (ниша, база, охваты, цель)
- **3 окна продаж** — закрытое для тёплой базы → бесплатный марафон/вебинар → финальное окно в блоге с бонусом
- **Декомпозиция цели** — средний и максимальный сценарии, с честным вычетом налога и процента отдела продаж (показываем эксперту чистую выручку, а не «валовую»)
- **Матрица «точка входа × тариф»** — три тарифа (базовый / стандарт / премиум) с разными ценами в зависимости от того, когда человек принял решение. Раннее решение — дешевле
- **Фишки продукта** — онбординг-созвон, оферта от проверенного юриста, опционально бонусный модуль монетизации
- **Условия сотрудничества** — шкала процента от чистой прибыли (20% / 25% / 30%)
- **Зоны ответственности** — что берём на себя (техничка, тексты, сценарии, лендинги, скрипты ОП), что остаётся эксперту (снимать по сценариям, провести события)
- **Таймлайн неделями, не датами** — стандартный темп 7 недель от подписания до старта потока. Discovery укладывается в 1 неделю, уже на 3-й идёт основной прогрев
- **Перспектива** — что делаем после первого запуска (автоворонка, холодный трафик, развитие в онлайн-школу)

### Как использовать

```
/presale
```

Команда задаст минимум вопросов (имя блогера, ниша, база, цель, условия) и соберёт документ. Сохраняется в `projects/_presale/blogger-[имя]-pitch.md` — отдельной папке, чтобы pre-sale черновики не мешались с реальными проектами.

### Что происходит, если клиент соглашается

Вся pre-sale работа → ни к чему не обязывает на стороне системы. Если сделка закрыта:

1. `/new-project` → создаём полноценный проект
2. `projects/_presale/blogger-[имя]-pitch.md` можно перенести в `projects/blogger-[имя]/brief/initial-pitch.md` как память о стартовой концепции
3. Работа идёт по обычному циклу — /intake → /audience → и т.д.

Методология скилла: [`.claude/skills/presale-strategy/SKILL.md`](.claude/skills/presale-strategy/SKILL.md)

## Быстрый старт

### 0. (Опционально) Собери pre-sale стратегию для потенциального клиента

Если эксперт ещё не подписан — сначала закрой сделку. Из корня репозитория:

```
/presale
```

Claude соберёт документ для отправки эксперту. После согласия — переходи к шагу 1.

Если эксперт уже подписан — пропусти этот шаг.

### 1. Заведи нового блогера

```
/new-project
```

Claude задаст 8 вопросов (имя, ниша, тип, база, соцсети, формат сотрудничества, deadline, ограничения) и создаст папку `projects/blogger-[имя]/` со скелетом проекта.

### 2. Получи материалы от эксперта

Попроси эксперта заполнить:
- `projects/blogger-[имя]/brief/client-intake.md` — анкета (версия hard или soft по типу ниши)
- `projects/blogger-[имя]/brief/discovery-call.md` — транскрипт разговора
- `projects/blogger-[имя]/brief/self-perception.md` — как эксперт сам видит ЦА
- `projects/blogger-[имя]/brief/social-links.md` — соцсети + размер базы
- `projects/blogger-[имя]/brief/verbal-numbers.md` — устно озвученные цифры

Если есть публичные сигналы (топ-посты, комменты, контент конкурентов) — в `brief/public-signals/`.

### 3. Перейди в папку проекта

```bash
cd projects/blogger-[имя]
```

И начни работу. Claude будет читать локальный CLAUDE.md и state.md вместо корневого.

### 4. Запусти первого агента

```
/intake
```

Он прочитает весь `brief/` и соберёт `research/project-digest.md` — источник правды для всех остальных агентов.

### 5. Дальше — `/next` ведёт по плану

```
/next
```

Орchestrator определяет, кого звать дальше, и запускает нужного агента. На ключевых точках (client-review) остановится и попросит эксперта утвердить артефакт перед продолжением.

### Полный поток

```
Onboarding  →  /intake
Discovery   →  /audience  →  /castdev (если есть база)  →  /audience (Проход 2)
Strategy    →  /product   →  /strategy (Проход 1 → эксперт выбирает формат → Проход 2)
Preparation →  /event, /tg, /ig, /reels, /landing (параллельно по зонам)
Execution   →  публикация по плану, ежедневные отчёты в execution-log.md
```

## Как устроена система

```
blogger-launch-system/
├── .claude/
│   ├── agents/         ← 11 агентов, каждый под свою задачу
│   ├── skills/         ← 6 методологических файлов
│   ├── commands/       ← slash-команды
│   └── system-map.md   ← визуальная карта системы
├── _TEMPLATE/          ← скелет для нового проекта (копируется /new-project)
├── projects/           ← конкретные проекты блогеров (твой рабочий фронт)
│   └── blogger-[name]/
│       ├── CLAUDE.md   ← инструкции под конкретного эксперта
│       ├── state.md    ← статус фаз, event log
│       ├── brief/      ← сырой вход от клиента
│       ├── research/   ← внутренние материалы (digest, hypotheses, briefs)
│       ├── data/       ← утверждённые артефакты (profile, matrix, strategy)
│       └── content/    ← готовый контент (tg/, ig/, webinar/, landings/)
└── CLAUDE.md           ← корневая инструкция системы
```

**Золотое правило:** работа над конкретным блогером ведётся **из папки проекта** (`cd projects/blogger-[имя]`), а не из корня. Тогда Claude подхватит нужный CLAUDE.md и не полезет в данные других блогеров.

### Source of truth

- `state.md` проекта — единственный источник правды о текущем статусе. Обновляется после каждого шага.
- `research/project-digest.md` — компактный источник правды о блогере. Все агенты (кроме `intake-analyst`) читают его вместо сырого `brief/`.
- `data/audience-profile.md`, `data/product-matrix.md`, `data/launch-strategy.md` — 3 обязательно утверждаемых экспертом артефакта.

## Фазы запуска

| # | Фаза | Владелец | Что происходит | Ключевой артефакт |
|---|---|---|---|---|
| 1 | **Onboarding** | intake-analyst | Парсинг брифа, сборка digest | `research/project-digest.md` |
| 2 | **Discovery** | audience-analyst + касдев-прогрев | Гипотезы сегментов → касдев (если есть база) → финальный профиль | `data/audience-profile.md` |
| 3 | **Strategy** | product-architect → launch-strategist | Оффер, стратегия, таймлайн, юнит-экономика | `data/product-matrix.md`, `data/launch-strategy.md` |
| 4 | **Preparation** | warmers + writers | Контент: прогревы, вебинар, лендинги | `content/` (всё) |
| 5 | **Execution** | продюсер | Публикация, ежедневные отчёты | `data/execution-log.md` |

### Статусы в state.md

- `pending` — не начато
- `in-progress` — агент работает
- `my-review` — ждёт ревью продюсера
- `client-review` — ждёт ревью эксперта
- `done` — завершено
- `blocked` — заблокировано, нужен ввод
- `skipped` — пропущено

### Ветвление

**По наличию базы (`has_base`):**
- `yes` — есть TG/IG с подписчиками → Discovery с касдевом
- `no` — базы нет → Discovery без касдева (профиль строится только из digest)

**По модели запуска (`launch_model`):**
- `warm` — есть база, трафик не нужен → только прогрев
- `cold` — базы нет → + трафик-план + лид-магнит + холодный лендинг
- `hybrid` — обе ветки параллельно

## Правила работы

### Для продюсера

1. **Всегда работай из папки проекта**, не из корня. `cd projects/blogger-[имя]` — и только потом запускай Claude или команды.
2. **Не параллель агентов.** Даже если кажется, что TG и IG warmer могли бы работать одновременно — по правилу системы они идут последовательно. Это осознанный компромисс: ты хочешь читать и проверять каждый артефакт отдельно.
3. **Client-review — не формальность.** Три артефакта (audience-profile, product-matrix, launch-strategy) обязательно должны быть утверждены экспертом, иначе последующие агенты построят на шатком фундаменте.
4. **State.md — единственная правда.** Не дублируй статус в других местах. Все агенты смотрят только туда.
5. **Brief/ может быть неполным на старте.** Intake-analyst пометит критичные пробелы → вызовешь research-scout и принесёшь недостающие публичные сигналы.

### Про tone-of-voice

В `projects/blogger-[имя]/CLAUDE.md` есть раздел «Бренд эксперта» — тон, стоп-слова, фирменные обороты. Заполни его перед запуском writers, иначе они будут писать нейтрально.

Если у тебя есть 5–10 живых постов/эфиров эксперта — положи их в `research/expert-voice.md`. Warmers прочитают и откалибруют голос.

## Частые вопросы

**Q: Зачем агенты вместо одного большого промпта?**
Каждый агент работает в изоляции — у него свой контекст, свои инструменты, своя модель. Это экономит токены (не читаешь всё сразу) и уменьшает ошибки (агент не путается в соседних задачах).

**Q: Почему Claude Pro, а не API?**
Система заточена под Pro — модели не выше Sonnet, нет параллелизма. На API будет быстрее и дешевле на больших объёмах, но требует настройки.

**Q: Можно ли пропустить касдев?**
Да, если `has_base = no` — касдев физически невозможен, система сама пропускает. Если есть база, но касдев не хочешь делать — ответы на анкету можно собрать другими способами (DM, созвоны) и положить в `brief/public-signals/castdev-responses.md`.

**Q: Что если эксперт не согласен с артефактом?**
На client-review эксперт даёт список правок. Продюсер передаёт их обратно автору-агенту через adhoc-режим (`/audience` с задачей «переделай раздел X по правкам эксперта»). После правок артефакт снова на client-review.

**Q: Можно ли использовать систему для нескольких блогеров параллельно?**
Да. Каждый проект в своей папке `projects/blogger-[имя]/` изолирован. Агенты одного проекта не читают данные другого (это явно запрещено в их промптах).

**Q: Что если агент ошибся / написал плохо?**
Проверь две вещи: (1) все ли входные артефакты утверждены и непротиворечивы, (2) заполнен ли раздел «Бренд эксперта» в CLAUDE.md. Часто проблема в сырых данных, не в агенте. Если данные ок — запусти агента повторно в adhoc-режиме с уточнением задачи.

**Q: Как обновить digest, если клиент принёс новые материалы?**
Скажи Claude «бриф обновился» или запусти `/intake` — intake-analyst пересоберёт digest. Orchestrator сам это заметит через сравнение дат модификации.
